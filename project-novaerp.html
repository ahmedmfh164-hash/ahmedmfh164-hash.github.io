NovaERP

RoleSolo developer

Why this project

DVLD and BMS taught me the fundamentals — schemas, validation, OOP — but both were desktop-bound: a WinForms client or a console session talking directly to its own data layer. That doesn't reflect how most production backends actually work. NovaERP is my attempt to close that gap: a system exposed as a web API, consumed by clients it doesn't know or control, with a domain model that has to stay correct regardless of what's calling it.

The scope is deliberately broad — an ERP touches inventory, orders, and accounts, all of which depend on each other — because that's exactly the kind of interlocking complexity I wanted to practice managing.

The architecture decision

I chose Clean Architecture specifically because of that interlocking complexity. With DVLD's Three-Tier setup, business logic still sat close enough to the data layer that the two were easy to tangle. Clean Architecture pushes that separation further by inverting the dependency direction entirely:

Domain → entities & business rules — zero external dependencies
Application → use cases, interfaces, DTOs — depends only on Domain
Infrastructure → EF Core, SQL Server — implements Application's interfaces
API → ASP.NET Core controllers — the thinnest layer of all

The point of the inward dependency rule is that Domain never imports anything. If I swapped SQL Server for PostgreSQL tomorrow, or replaced the API layer with a gRPC service, the business rules wouldn't need to change at all — only Infrastructure and API would know the difference.

What's actually built vs. planned

I'm being specific here because the project is mid-build: the four-layer skeleton is in place, along with EF Core's code-first model and the initial migration against SQL Server. What's still ahead is the part that matters most — the actual domain logic for inventory and order management, the use cases that enforce business rules across them, and integration tests that prove the layers actually stay decoupled under real conditions.

I'd rather show the architecture honestly at this stage than backfill a feature list that overstates where it is.

What I'm doing next

Three things, roughly in order: finish the inventory module end-to-end as a proof of the architecture, add a unit test suite around the Application layer so the use cases are verifiable independent of the database, and document the emerging API surface with Swagger as endpoints stabilize.
