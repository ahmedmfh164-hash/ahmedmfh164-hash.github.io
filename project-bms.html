BMS

A console banking application focused on the part every bank interface depends on underneath: a balance and history that stay correct no matter what order things happen in.

RoleSolo developer
StackC++, OOP, File I/O
DurationMay — Jun 2026
Sourcegithub ↗

WHY STRIP AWAY THE INTERFACE

An app, an ATM, a teller window — every banking front end is a different skin on the same requirement: balances and history have to be accurate and recoverable regardless of the order operations happen in. BMS deliberately has no real interface beyond a console, because the point was to get that underlying requirement right without a UI distracting from it.

MODELING ACCOUNTS AS OBJECTS, NOT ROWS

The core of the system is two related objects:

Account
accountIDint
ownerNamestring
balancedouble
Transaction
accountIDref
typedeposit/withdraw
amountdouble

Encapsulation means balance is never touched directly — only account methods can change it, so every mutation passes through validation. Inheritance and abstraction shape how account behavior is organized, so common logic doesn't get duplicated if more account types are added later.

PERSISTENCE WITHOUT A DATABASE

A console-scale project didn't need a database server, so BMS uses file handling instead — but the goal was the same guarantee a real system needs:

On exit → serialize accounts & transaction log to file
On launch → read file back into memory before accepting input

That keeps the project dependency-free while still proving out "never lose a record" — the same requirement a production system would need from its actual database layer.

WHAT I'D DO DIFFERENTLY NOW

Flat-file storage works for a console demo, but it has no real transaction safety — a write can be interrupted half-done. Swapping it for SQLite would be the natural next step: dependency-light, but with commits that can't be left in a broken state, which a banking system should have even at this scale.
